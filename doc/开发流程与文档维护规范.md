# 开发流程与文档维护规范

## 📋 目的
确保每次新增功能、修改现有功能或修复重要 bug 时，相关技术文档都能得到及时更新，保持文档与代码同步。

## 🎯 适用范围
- 新增功能模块
- 修改现有功能
- 数据库结构变更
- API 接口变更
- 技术栈更新
- 重要 bug 修复

## 📝 文档维护清单

### 1. 需求/功能新增时的文档更新

#### 必须检查的文档：
- [ ] **博客后端设计文档.md** - 更新功能模块设计
- [ ] **技术栈和约束.md** - 如涉及技术栈变更
- [ ] **API 文档** - 在 Swagger/OpenAPI 中更新接口说明
- [ ] **数据库迁移脚本** - 如涉及数据库变更

#### 可选更新的文档：
- [ ] README.md - 如影响用户使用
- [ ] 部署文档 - 如影响部署流程

---

## 🔧 执行流程

### 步骤 1：开发前准备
在开始开发新功能前：
1. 评估功能影响的范围
2. 确定需要更新的文档清单
3. 在任务管理系统中创建文档更新任务

### 步骤 2：开发过程中
1. **编写代码时**：
   - 同步更新 Swagger 注解（`@Operation`, `@ApiResponse` 等）
   - 确保类和方法注释完整
   
2. **数据库变更时**：
   - 创建 Flyway 迁移脚本
   - 在设计文档中更新 ER 图和表结构说明

3. **接口变更时**：
   - 更新 Controller 中的 Swagger 注解
   - 记录接口变化（请求参数、响应格式）

### 步骤 3：提交代码前检查清单

#### ✅ 代码检查
- [ ] 代码编译通过
- [ ] 单元测试通过
- [ ] 代码符合规范
- [ ] 注释完整清晰

#### ✅ 文档检查
- [ ] Swagger 文档已更新（访问 `/swagger-ui.html` 检查）
- [ ] 设计文档已更新（如有新功能）
- [ ] 技术栈文档已更新（如有技术变更）
- [ ] 数据库变更已记录（如有）

### 步骤 4：Git 提交规范

提交信息格式：
```
<type>(<scope>): <subject>

<body>

[文档更新] 
- 更新了 XXX 功能说明
- 添加了 XXX 接口文档
```

示例：
```
feat(posts): 添加文章搜索功能

- 实现基于关键词的文章搜索
- 支持标题和内容模糊匹配
- 添加搜索结果分页

[文档更新]
- 更新博客后端设计文档.md - 2.2.4 文章查询章节
- 更新 Swagger API 文档 - POST /api/v1/posts/search
```

## 📚 各文档更新指南

### 1. 博客后端设计文档.md

**何时更新**：
- 新增功能模块
- 修改现有功能逻辑
- 添加新的业务流程

**更新位置**：
- 第 2 章 功能模块设计 - 对应模块下添加内容
- 第 3 章 数据库设计 - 如有表结构变更
- 第 4 章 接口设计 - 添加新接口

**示例**：
```markdown
### 2.2.6 文章搜索（新增）
- 支持关键词搜索（标题、内容）
- 支持组合筛选（作者、分类、标签）
- 搜索结果高亮显示
- 分页展示
```

### 2. 技术栈和约束.md

**何时更新**：
- 引入新的依赖库
- 更换技术组件
- 更新版本规范

**更新位置**：
- 开发技术栈章节
- 开发规范章节

**示例**：
```markdown
# 开发技术栈
语言：Java 17
框架：Spring boot 3.2.4, Spring Security, MybatisPlus
【新增】搜索引擎：Elasticsearch 8.x
```

### 3. Swagger/OpenAPI 文档

**何时更新**：
- 新增 API 接口
- 修改接口参数
- 变更响应格式

**更新方式**：
在 Controller 方法上更新注解：

```java
@Operation(summary = "搜索文章", description = "根据关键词搜索文章内容")
@PostMapping("/search")
public Response<List<Post>> searchPosts(
    @RequestBody PostSearchRequest request) {
    // ...
}
```

### 4. 数据库迁移脚本

**何时更新**：
- 新增表
- 修改表结构
- 添加索引

**位置**：
`src/main/resources/db/migration/`

**命名规范**：
```
V{版本号}__{描述}.sql
例如：V1_0_1__add_post_search_index.sql
```

## 🚀 自动化辅助

### Git Hook 检查（推荐配置）

在 `.git/hooks/pre-commit` 中添加检查脚本：

```bash
#!/bin/bash

# 检查是否修改了 Controller 文件
if git diff --cached --name-only | grep -q "Controller\.java"; then
    echo "⚠️ 检测到 Controller 变更，请确认是否已更新 Swagger 文档"
    read -p "是否继续提交？(y/n): " confirm
    if [ "$confirm" != "y" ]; then
        exit 1
    fi
fi

# 检查是否创建了新的迁移脚本
if git diff --cached --name-only | grep -q "db/migration/V.*\.sql"; then
    echo "✅ 检测到数据库迁移脚本"
    echo "📝 请记得在博客后端设计文档.md 中更新数据库设计章节"
fi
```

### IDE 模板（IntelliJ IDEA）

创建 Live Template，快速生成文档更新记录：

**模板缩写**：`docupdate`
**模板内容**：
```
[文档更新]
- [ ] 更新博客后端设计文档.md
- [ ] 更新 Swagger API 文档
- [ ] 更新技术栈文档（如适用）
- [ ] 添加数据库迁移脚本（如适用）
```

- [ ] 设计文档是否覆盖所有功能
- [ ] 是否有未记录的临时方案

- [ ] 审查文档完整性
- [ ] 更新过时的技术说明
- [ ] 补充示例代码和截图

## 🎓 最佳实践

1. **文档即代码**：将文档视为代码的一部分，同等重要
2. **小步快跑**：每次变更只更新相关文档，避免积压
3. **自动化优先**：能用注解生成的文档（如 Swagger）不手动维护
4. **版本对应**：文档版本与代码版本保持一致
5. **审查机制**：Code Review 时必须包含文档审查

## ⚠️ 违规处理

如发现以下情况，需在团队会议中复盘：
- 功能已上线但文档未更新
- 文档与实际功能严重不符
- 多次提醒仍未更新文档

**改进措施**：
1. 完善检查清单
2. 加强 Code Review
3. 建立文档质量指标
