# 📚 开发流程与文档维护指南

本目录包含项目开发和文档维护的相关规范和模板。

---

## 📁 文件说明

### 核心规范
- **开发流程与文档维护规范.md** - 主要规范文档，定义了整个开发流程和文档更新要求
- **功能开发文档更新检查清单.md** - 每次开发新功能时的检查清单模板

### 设计文档
- **博客后端设计文档.md** - 系统功能设计、数据库设计、接口设计
- **技术栈和约束.md** - 技术栈说明和开发规范

## 🚀 快速开始

### 新增功能时

1. **阅读规范**：先查看 `开发流程与文档维护规范.md`
2. **复制清单**：复制 `功能开发文档更新检查清单.md` 到新文件
3. **执行检查**：按照清单逐项完成并勾选
4. **规范提交**：使用 `GIT_COMMIT_TEMPLATE.md` 中的模板编写提交信息

### 修改现有功能时

1. **评估影响**：确定影响的范围和文档
2. **更新文档**：
   - Swagger 注解（必须）
   - 设计文档相关章节（如适用）
   - 技术栈文档（如涉及技术变更）
3. **提交代码**：在提交信息中包含 `[文档更新]` 标记

---

## ✅ 最简流程（小功能）

如果功能较小，至少完成以下 4 项：

1. ✅ **Swagger 注解** - 所有 Controller 方法添加完整注解
2. ✅ **设计文档** - 在 `博客后端设计文档.md` 中更新对应章节
3. ✅ **数据库脚本** - 如有数据库变更，创建 Flyway 迁移脚本
4. ✅ **提交信息** - 包含 `[文档更新]` 说明

---

## 📋 详细流程（大功能）

## 📊 检查机制

### 开发自查
- 每次提交前使用检查清单
- 确保文档与代码同步

### Code Review
- 必须包含文档审查
- 使用检查清单作为 Review 依据

### 定期检查
- 每周：Swagger 文档健康度检查
- 每月：设计文档完整性审查

---

## 💡 最佳实践

1. **文档即代码** - 将文档视为代码的一部分
2. **小步快跑** - 每次变更立即更新文档，不积压
3. **自动化优先** - 能用注解生成的不手动维护
4. **版本对应** - 文档版本与代码版本保持一致
5. **审查机制** - Code Review 必须包含文档审查

---

## ⚠️ 常见错误

❌ **错误做法**：
- 功能上线后才补文档
- 只更新代码，不更新文档
- 文档更新不及时，与实际功能不符
- Swagger 注解不完整或过时

✅ **正确做法**：
- 开发过程中同步更新文档
- 文档更新作为完成的定义（DoD）一部分
- 使用检查清单确保不遗漏
- 定期审查文档质量
